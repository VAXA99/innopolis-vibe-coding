# Изменения за 18 апреля 2026

Краткое описание того, что было сделано сегодня в репозитории и связанных сервисах (iOS‑приложение «будильник», backend на Render, Supabase).

## iOS (папка `будильник/`)

### Регистрация и аккаунт
- Уточнена логика показа экрана регистрации при первом входе.
- Добавлен сценарий **«Выйти из аккаунта»** в настройках: сброс локальных данных профиля и возврат на экран регистрации.
- При выходе дополнительно очищаются «облачные» настройки Huawei‑синка (endpoint/token и связанные значения), чтобы не тянуть данные предыдущего пользователя.

### Supabase (клиентская часть без SDK)
- В `Info.plist` заполнены `SUPABASE_URL` и `SUPABASE_ANON_KEY` для PostgREST‑вызовов.
- После регистрации сохраняется дублирующая запись в `app_registrations` (как было задумано ранее).
- Добавлен модуль `SupabaseProfiles.swift` для работы с таблицей `profiles` через PostgREST:
  - загрузка профиля по `id`
  - создание профиля при отсутствии
  - обновление `paid` (для MVP‑демо)

### Paywall (оплата) и доступ к контенту
- Добавлен paywall‑экран в стиле приложения.
- Логика доступа: если `profiles.paid = false`, показывается paywall; если `true` — основной контент.
- Кнопка оплаты открывает страницу успеха на backend: `/success` с параметром `return_to`, который ведёт обратно в приложение через deep link.
- Добавлен URL scheme `smartalarm://` и обработка возврата из браузера в приложение.

### Microsoft Clarity
- Подключён пакет `microsoft/clarity-apps` (SwiftPM) и добавлена инициализация Clarity при старте приложения (через `#if canImport(Clarity)`).

### Мелкие исправления сборки
- Исправлен вызов `UIApplication.shared.open` на async‑вариант с `await` (ошибка компиляции Swift).

## Backend (`huawei-sync-server/` на Render)

### Новый маршрут для оплаты
- Добавлен `GET /success`: HTML‑страница «Оплата успешна» + кнопка возврата в приложение по deep link + авто‑переход.

## Supabase (облако, не в git)

> Эти изменения применялись в проекте Supabase и не обязаны отражаться в git‑истории, но они влияют на работу приложения.

- Создана таблица `public.profiles` с полями `id (uuid, PK)`, `email (text)`, `paid (boolean, default false)`.
- Для MVP‑сценария с PostgREST на anon‑ключе добавлены RLS‑политики на `profiles` (select/insert/update для `anon`).
  - Это **временная** конфигурация для демонстрации; для продакшена обычно переходят на Supabase Auth и политики вида `auth.uid() = id`.

## Git / ветки

- Изменения были запушены в `main`.
- `dev` был синхронизирован с `main` (fast‑forward), чтобы Render, привязанный к `dev`, получил те же коммиты.

## Что нужно сделать вручную после пуша

- В Render выполнить деплой **последнего коммита** ветки, которую реально деплоит сервис (обычно `dev`), и проверить, что `GET /success` открывается (не 404).
